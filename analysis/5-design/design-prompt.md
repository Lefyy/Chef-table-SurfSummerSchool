/// Нейросеть - Codex (Cloud версия бесплатно).

Ты — senior AI analyst / solution analyst. Твоя задача — на основе предоставленных артефактов строго и аккуратно описать API sequence diagram и модель данных с ERD для приложения.

## Источники

Используй только эти материалы:

1. `analysis/2-elicitation/domain-description.md`
2. `analysis/3-requirements/business-requirements/business-requirements.md`
3. `analysis/3-requirements/functional-requirements/functional-requirements.md`
4. `analysis/3-requirements/non-functional-requirements/non-functional-requirements.md`
5. `analysis/3-requirements/user-stories/user-stories.md`
6. `analysis/3-requirements/use-cases/use-cases.md`
7. `analysis/4-design-brief/` — **основной источник истины**

## Жесткие правила

* Если источники противоречат друг другу, приоритет всегда у `analysis/4-design-brief/`.
* Не выдумывай сущности, поля, статусы, роли, сценарии и API-операции, которых нет в источниках.
* Если в источниках есть пробелы или неоднозначности, фиксируй их как `Assumptions` или `Open questions`, но не заполняй фантазией.
* Термины, названия сущностей, статусы и действия должны быть согласованы со всеми документами, особенно с design brief.
* Описывай только то, что реально нужно для реализации и проектирования.
* Пиши кратко, структурно и технично.
* Не показывай черновые рассуждения, внутреннюю логику или chain-of-thought. Покажи только итоговый результат.

## Что нужно сделать

Сформируй **ровно 2 Markdown-файла**:

### 1) `api-sequece.md`

Документ должен описывать sequence diagram API приложения.

Содержимое:

* Краткое назначение документа.
* Контекст и границы системы.
* Список ключевых сценариев, которые нужно отразить в sequence diagram.
* Для каждого сценария:

  * участники / lifelines;
  * триггер;
  * основные шаги вызовов API;
  * синхронные и асинхронные взаимодействия, если они есть;
  * валидации, авторизация, ошибки, retry / rollback, если это отражено в источниках;
  * результат сценария.
* Диаграммы должны быть оформлены в `Mermaid sequenceDiagram`.
* Если сценариев несколько, разделяй их подзаголовками.
* Если есть важные альтернативные ветки, покажи их явно.
* Не добавляй лишнюю архитектуру, которой нет в источниках.

Рекомендуемая структура файла:

* `# API Sequence`
* `## Scope`
* `## Key scenarios`
* `### Scenario 1 ...`
* `Mermaid`-блоки sequence diagram
* `## Assumptions`
* `## Open questions`

### 2) `data-model.md`

Документ должен описывать модель данных и ERD.

Содержимое:

* Краткое назначение документа.
* Список доменных сущностей.
* Для каждой сущности:

  * назначение;
  * ключевые атрибуты;
  * primary key;
  * foreign keys;
  * уникальные ограничения;
  * nullable / mandatory поля, если это известно;
  * комментарии по жизненному циклу и статусам, если применимо.
* Все связи между сущностями:

  * кардинальность;
  * обязательность;
  * направленность;
  * смысл связи.
* Диаграмма ERD должна быть оформлена в `Mermaid erDiagram`.
* Если уместно, добавь отдельный раздел с правилами целостности данных.
* Не смешивай модель данных с UI-логикой.
* Не добавляй сущности без подтверждения в источниках.

Рекомендуемая структура файла:

* `# Data Model`
* `## Scope`
* `## Entities`
* `## Relationships`
* `## ERD`
* `## Integrity rules`
* `## Assumptions`
* `## Open questions`

## Метод работы

1. Сначала внимательно извлеки доменные термины, роли, сценарии, сущности, состояния и ограничения из всех источников.
2. Затем сопоставь use cases, user stories и functional requirements с design brief.
3. После этого построй:

   * последовательности API-взаимодействий для ключевых сценариев;
   * ERD и словарь сущностей.
4. Убедись, что диаграммы и текст описывают одну и ту же систему без противоречий.
5. Если данных недостаточно, остановись на уровне точных допущений, а не домыслов.

## Формат ответа

Верни только два блока Markdown, без лишнего текста:

1. содержимое файла `api-sequece.md`
2. содержимое файла `data-model.md`

Каждый блок должен быть готов к сохранению в отдельный `.md` файл.

## Качество результата

Перед финальным ответом проверь:

* все ключевые сценарии покрыты;
* сущности в ERD согласованы с текстом;
* нет противоречий с design brief;
* Mermaid-синтаксис корректен;
* нет выдуманных элементов;
* названия стабильны и единообразны.
