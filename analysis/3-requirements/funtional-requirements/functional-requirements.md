# Функциональные требования системы

## Краткое резюме

- Всего функциональных требований: 30
- Требований Must Have: 25
- Требований Should Have: 5
- Требований Could Have: 0
- Требований Won't Have: 0

## Функциональные требования

### FR-01 — Авторизация клиента по телефону и SMS-коду

| Атрибут | Значение |
|---|---|
| Приоритет (MoSCoW) | Must Have |
| Формулировка | Система должна позволять клиенту авторизоваться по номеру телефона и одноразовому SMS-коду. |
| Описание | Клиент вводит номер телефона, после чего система инициирует отправку одноразового SMS-кода через бэкенд. После ввода корректного кода система должна предоставить клиенту JWT/сессию. |
| Область | Авторизация, клиентский профиль |
| Связь с исходным описанием | Прямое |
| Источник | Описание домена: процесс «Вход / Авторизация»; бизнес-требование BR-02; бизнес-правило домена BR-10. |
| Примечания / неоднозначности | Интерфейсы администратора, владельца и шефа не входят в текущую поставку. |

### FR-02 — Проверка срока действия и числа попыток ввода SMS-кода

| Атрибут | Значение |
|---|---|
| Приоритет (MoSCoW) | Must Have |
| Формулировка | Система должна проверять, что SMS-код введён в течение 5 минут и не превышен лимит из 3 попыток ввода. |
| Описание | При вводе кода система должна передавать данные проверки в бэкенд и не предоставлять сессию, если код просрочен, неверен или лимит попыток исчерпан. |
| Область | Авторизация, проверки входных данных |
| Связь с исходным описанием | Прямое |
| Источник | Описание домена: процесс «Вход / Авторизация». |
| Примечания / неоднозначности | При ошибках бэкенда без более конкретного описания система должна отображать сообщение «Ошибка сервера». Повторная отправка SMS-кода описана в FR-03. |

### FR-03 — Повторная отправка SMS-кода

| Атрибут | Значение |
|---|---|
| Приоритет (MoSCoW) | Must Have |
| Формулировка | Система должна позволять клиенту повторно отправить SMS-код не ранее чем через 1 минуту после предыдущей отправки и не более 5 раз. |
| Описание | После отправки SMS-кода система должна блокировать повторную отправку на 1 минуту. По истечении 1 минуты клиент должен иметь возможность запросить новый SMS-код, если количество повторных отправок не превысило 5. |
| Область | Авторизация, SMS-код |
| Связь с исходным описанием | Уточнено заказчиком |
| Источник | Дополнительные уточнения заказчика: повторная отправка SMS-кода доступна через минуту после отправки; максимальное количество повторных отправок — 5. |
| Примечания / неоднозначности | После 5 повторных отправок система не должна отправлять новый SMS-код в рамках текущего сценария авторизации. |

### FR-04 — Получение данных из существующего бэкенда

| Атрибут | Значение |
|---|---|
| Приоритет (MoSCoW) | Must Have |
| Формулировка | Система должна получать из существующего бэкенда данные о слотах, программах, шефах, тарифах проката, бронях, отменах, аллергиях и оценках. |
| Описание | Клиентское приложение должно использовать существующий бэкенд как источник данных и результатов операций. Система не должна создавать или редактировать расписание, слоты, программы и шефов. |
| Область | Интеграция с бэкендом, границы системы |
| Связь с исходным описанием | Прямое |
| Источник | Описание домена: краткое резюме, роли, сущности и границы системы; бизнес-требование BR-03. |
| Примечания / неоднозначности | Изменения бэкенда и админки находятся вне объёма MVP. |

### FR-05 — Отображение расписания на ближайшие 7 дней по умолчанию

| Атрибут | Значение |
|---|---|

| Описание | При открытии расписания система должна показывать слоты, полученные из бэкенда, за период от текущей даты до конца седьмого календарного дня включительно без необходимости предварительно задавать фильтр дат. |
| Область | Расписание, поиск класса |
| Связь с исходным описанием | Прямое |
| Источник | Описание домена: процесс «Просмотр и фильтрация расписания»; бизнес-требование BR-07. |
| Примечания / неоднозначности | Период считается от текущей даты согласно уточнению заказчика. |

### FR-07 — Отображение пустого состояния расписания

|---|---|
| Приоритет (MoSCoW) | Should Have |

| Описание | При отсутствии подходящих классов система должна информировать клиента пустым состоянием «Пока нет доступных классов». |
| Область | Расписание, результаты фильтрации |
| Связь с исходным описанием | Прямое |
| Источник | Описание домена: процесс «Просмотр и фильтрация расписания», блок «Пустое состояние». |
| Примечания / неоднозначности | Приоритет Should Have выбран, так как сообщение не создаёт основной сценарий бронирования, но требуется для корректной обработки пустого результата. |

| Примечания / неоднозначности | При ошибках бэкенда без более конкретного описания система должна отображать сообщение «Ошибка сервера». |

### FR-12 — Выбор доступного проката экипировки при бронировании

| Примечания / неоднозначности | В исходном описании явно перечислены набор ножей и фартук; цена определяется бэкендом. |

### FR-13 — Указание аллергий при бронировании

| Атрибут | Значение |
|---|---|
| Приоритет (MoSCoW) | Must Have |
| Формулировка | Система должна позволять клиенту указать ноль, одну или несколько аллергий из справочника при оформлении брони. |
| Описание | Указание аллергий должно быть необязательным. При следующем бронировании система должна автоматически подставлять аллергии, ранее сохранённые в профиле клиента, с возможностью использовать их в данных новой брони. |
| Область | Бронирование, аллергии |
| Связь с исходным описанием | Прямое |
| Источник | Описание домена: сущность «Аллергия», процесс «Воронка бронирования», бизнес-правила BR-06 и BR-12; бизнес-требование BR-10. |
| Примечания / неоднозначности | Добавление аллергии вне справочника, включая ввод свободным текстом, запрещено согласно уточнению заказчика. |

### FR-14 — Сохранение выбранных аллергий в профиле клиента

| Атрибут | Значение |

| Описание | Аллергии, указанные клиентом, должны быть доступны в разделе профиля «Мои аллергии» и автоматически подставляться при следующем бронировании. |
| Область | Профиль клиента, аллергии |
| Связь с исходным описанием | Прямое |
| Источник | Описание домена: сущности «Аллергия» и «Профиль клиента», бизнес-правило BR-06; бизнес-требование BR-10. |
| Примечания / неоднозначности | Автоподстановка сохранённых аллергий при следующем бронировании добавлена по уточнению заказчика. |

### FR-18 — Обработка невозможности бронирования при отсутствии мест

| Описание | Если `availableSeats = 0` или бэкенд отказал в создании брони из-за отсутствия мест, клиентское приложение должно показать ошибку отсутствия мест и перенести пользователя на страницу листинга классов. |

| Связь с исходным описанием | Выведено из бизнес-правил |
| Источник | Описание домена: бизнес-правило BR-01; бизнес-требования BR-04 и BR-05. |
| Примечания / неоднозначности | Поведение при отсутствии мест уточнено заказчиком. Для иных ошибок бэкенда без конкретного описания используется сообщение «Ошибка сервера». |

| Атрибут | Значение |

## Неясности и пробелы в исходном описании

- Нерешённых неясностей после уточнений заказчика не осталось.
- Правила возвратов, компенсаций или альтернатив при отмене класса студией находятся вне скоупа текущей поставки.
- Функциональные сценарии переносов брони, листа ожидания, штрафов, онлайн-оплаты, промокодов, подписок, уведомлений о новых слотах, лояльности и реферальных программ исключены из MVP.

## Противоречия и дубли

- В исходных материалах статус отмены студией формулируется как «Отменён студией» и «Отменена студией»; в требованиях использована формулировка «Отменена студией», но требуется терминологическое уточнение.
- Требования об отсутствии двойных броней, наличии свободных мест и атомарном создании брони пересекаются. В функциональных требованиях они разделены на создание брони через бэкенд и обработку отказа при отсутствии мест.
- Пуш-уведомления упоминаются в контексте отмены класса студией, но в описании домена явно исключены из MVP; функциональное требование на push, email или SMS-уведомления не включено.
- Оплата на месте описана как бизнес-правило и граница MVP. В функциональные требования включено только отображение статуса оплаты со значениями «Не оплачено» и «Оплачено»; онлайн-оплата и связанные платёжные сценарии исключены из MVP.
