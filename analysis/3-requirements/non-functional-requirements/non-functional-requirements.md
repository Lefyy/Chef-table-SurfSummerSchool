# Нефункциональные требования системы

## Краткое резюме

* Система: адаптивное веб-приложение для записи клиентов на кулинарные классы Surf Summer School Chef Table.
* Масштаб бизнеса: 1 физическая точка, небольшое расписание классов, оплата на месте, готовый бэкенд является источником истины.
* Всего нефункциональных требований: 16.
* Приоритеты MoSCoW для NFR-01–NFR-16: Must Have — 10, Should Have — 2, Could Have — 4, Won't Have — 0. Дополнительно задано Should Have-требование к тестовому окружению.
* Все ранее неясные пункты зафиксированы как конкретные целевые значения и правила приемки.

## Принятые допущения для расчетов

| Код | Допущение |
| --- | --- |
| A-NFR-01 | В MVP обслуживается одна площадка и один публичный клиентский веб-интерфейс. |
| A-NFR-02 | Нормальная нагрузка: до 20 одновременных активных пользователей; пиковая нагрузка: до 50 одновременных активных пользователей в течение 15 минут после публикации расписания или рекламной рассылки. |
| A-NFR-03 | Типовой объем расписания: до 40 классов на ближайшие 7 дней; верхний предел для проектирования MVP: 100 классов на ближайшие 7 дней. |
| A-NFR-04 | Один класс содержит 8 или 12 мест, поэтому пиковое число операций бронирования для одной недели ограничено бизнес-масштабом одной точки. |
| A-NFR-05 | Персональные данные клиентов хранятся только в объеме, необходимом для авторизации, бронирования, учета аллергий, истории посещений и обратной связи. |

## Формат требований

Каждое требование описывается в едином формате:

* **Приоритет** — MoSCoW: Must Have, Should Have или Could Have.
* **Метрика / правило** — измеримое целевое значение или обязательное правило поведения.
* **Приемка** — как проверить выполнение требования.
* **Обоснование** — почему значение подходит для небольшого бизнеса с одной точкой.

## Нефункциональные требования

### NFR-01 — Производительность пользовательских сценариев

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | При нормальной нагрузке p95 времени ответа пользовательского интерфейса после получения ответа бэкенда: открытие расписания — не более 2 секунд; применение фильтра по типу кухни, дате, свободным местам или прокату — не более 1 секунды; открытие карточки класса — не более 1 секунды; отправка бронирования, отмены бронирования и оценки — не более 3 секунд без учета времени ответа SMS-шлюза. |
| Приемка | Нагрузочный или интеграционный прогон на тестовом окружении с 20 одновременными пользователями, расписанием из 40 классов и p95 по каждому сценарию не выше целевого значения. |
| Обоснование | Значения достаточны для небольшой студии: интерфейс остается быстрым для клиента, но не требует избыточной инфраструктуры. |

### NFR-02 — Производительность при пиковой нагрузке

| Атрибут | Значение |
| --- | --- |
| Приоритет | Should Have |
| Метрика / правило | При пиковой нагрузке до 50 одновременных активных пользователей в течение 15 минут p95 времени ответа для чтения расписания и карточек классов должен быть не более 4 секунд, а p95 для операций бронирования и отмены — не более 5 секунд при условии доступности бэкенда. |
| Приемка | Нагрузочный прогон с 50 виртуальными пользователями, 100 классами в расписании на 7 дней и смесью операций: 70% чтение, 20% фильтрация, 10% бронирование/отмена. |
| Обоснование | Пик соответствует малому бизнесу после публикации расписания или рекламы и не предполагает масштаб федерального сервиса. |

### NFR-03 — Доступность системы

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Целевая доступность публичного веб-приложения — не менее 99,0% в календарный месяц без учета заранее объявленных окон обслуживания и недоступности внешних провайдеров SMS. Максимальная длительность одного незапланированного простоя клиентского приложения — 2 часа. |
| Приемка | Ежемесячный отчет мониторинга доступности главной страницы и health-check endpoint с интервалом проверки не реже 1 раза в минуту. |
| Обоснование | 99,0% дает до 7 часов 18 минут простоя в месяц, что реалистично для MVP одной точки и не приводит к неоправданным затратам на high availability. |

### NFR-04 — Окна обслуживания и деградация при сбоях зависимостей

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Плановое обслуживание допускается не чаще 2 раз в месяц, не дольше 2 часов за одно окно, в период с 00:00 до 06:00 по локальному времени бизнеса. При недоступности бэкенда система должна показывать понятное сообщение и не создавать локальные бронирования. При недоступности SMS-шлюза система должна сообщать, что вход временно невозможен, и предлагать повторить попытку позже. |
| Приемка | Проверка сценариев с отключенным бэкендом и SMS-зависимостью на тестовом окружении; отсутствие неконсистентных локальных операций. |
| Обоснование | Для малого бизнеса важнее предсказуемое поведение и отсутствие неверных броней, чем сложная автономная работа. |

### NFR-05 — Масштабируемость по пользователям и данным

| Атрибут | Значение |
| --- | --- |
| Приоритет | Should Have |
| Метрика / правило | Архитектура MVP должна поддерживать без изменения пользовательских сценариев: до 50 одновременных активных пользователей, до 100 классов в расписании на 7 дней, до 5 000 зарегистрированных клиентов, до 20 000 записей истории бронирований за период хранения. |
| Приемка | Проверка тестовыми данными указанного объема: расписание открывается, фильтры применяются, профиль и история клиента доступны в пределах требований NFR-01 и NFR-02. |
| Обоснование | Запас покрывает рост одной точки без преждевременной оптимизации под сеть филиалов. |

### NFR-06 — Ограничение масштабирования за пределами MVP

| Атрибут | Значение |
| --- | --- |
| Приоритет | Could Have |
| Метрика / правило | MVP не обязан поддерживать несколько филиалов, разные часовые пояса площадок, франшизную модель, отдельные расписания по городам и автоматическое горизонтальное масштабирование под сотни одновременных пользователей. |
| Приемка | В документации и backlog такие возможности отмечены как будущие расширения, а не как критерии приемки MVP. |
| Обоснование | Бизнес работает в одной точке, поэтому дополнительные уровни масштабирования увеличат сложность без немедленной пользы. |

### NFR-07 — Резервное копирование и восстановление

| Атрибут | Значение |
| --- | --- |
| Приоритет | Could Have |
| Метрика / правило | Для данных, источником истины которых является бэкенд, должны выполняться ежедневные автоматические резервные копии с RPO не более 24 часов и RTO не более 8 рабочих часов. Резервные копии должны храниться не менее 14 календарных дней. |
| Приемка | Наличие регламента резервного копирования и успешная тестовая процедура восстановления не реже 1 раза в квартал. |
| Обоснование | Резервное копирование важно, но имеет низкий приоритет для MVP клиентского приложения, так как оно не является отдельным источником истины. |

### NFR-08 — Локальное хранение и восстановление клиентского приложения

| Атрибут | Значение |
| --- | --- |
| Приоритет | Could Have |
| Метрика / правило | Клиентское приложение не должно хранить персональные данные в localStorage, sessionStorage или IndexedDB дольше текущей авторизованной сессии. Восстановление данных после сбоя выполняется через повторное получение актуального состояния из бэкенда. |
| Приемка | Проверка браузерного хранилища после выхода из профиля и закрытия сессии; отсутствие телефона, аллергий, истории бронирований и комментариев в долгоживущем хранилище браузера. |
| Обоснование | Это снижает требования к резервному копированию фронтенда и уменьшает риск утечки данных с устройства клиента. |

### NFR-09 — Логирование

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Система должна вести структурированные логи для ключевых событий: запуск приложения, ошибки API, неуспешная авторизация, успешное создание брони, ошибка создания брони, отмена брони, отправка оценки, отказ внешней зависимости. Каждый лог должен содержать timestamp в ISO 8601, уровень, компонент, correlation_id, тип события и техническое описание без персональных данных. |
| Приемка | Ревью примеров логов и тестовое выполнение ключевых сценариев; в логах есть correlation_id, но нет полного номера телефона, SMS-кода, аллергий, комментариев клиента и платежных данных. |
| Обоснование | Логирование имеет повышенный приоритет, потому что поддержка малого сервиса должна быстро разбирать ошибки бронирования и авторизации. |

### NFR-10 — Метрики, мониторинг и алерты

| Атрибут | Значение |
| --- | --- |
| Приоритет | Could Have |
| Метрика / правило | Система должна собирать минимальный набор технических метрик: доступность, p95 времени ответа ключевых сценариев, доля 4xx/5xx ошибок API, число ошибок отправки SMS-кода, число ошибок создания брони. Алерт должен создаваться при недоступности приложения более 5 минут подряд или при доле 5xx выше 5% за 10 минут. |
| Приемка | Наличие dashboard или отчета мониторинга с указанными метриками и тестовый алерт в тестовом окружении. |
| Обоснование | Метрики полезны для эксплуатации, но имеют меньший приоритет, чем само логирование и корректность бронирования. |

### NFR-11 — Хранение персональных данных

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Персональные данные клиента, включая номер телефона, сохраненные аллергии, историю бронирований, оценки и комментарии, должны храниться не дольше 3 месяцев после последней активности клиента, если более длительное хранение не требуется законом или явным согласием клиента. По истечении срока данные должны быть удалены или обезличены. |
| Приемка | Проверка регламента удаления/обезличивания и тестовый сценарий для записей старше 3 месяцев. |
| Обоснование | Срок 3 месяца достаточен для повторных посещений и поддержки спорных ситуаций, но ограничивает накопление персональных данных. |

### NFR-12 — Нормативные требования к персональным данным

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Система должна обрабатывать персональные данные по принципам минимизации, целевого использования, ограничения срока хранения, защиты от несанкционированного доступа и удаления по истечении срока хранения. Для РФ необходимо учитывать требования 152-ФЗ «О персональных данных»: наличие политики обработки персональных данных, согласия на обработку при регистрации/авторизации, определенной цели обработки и порядка отзыва согласия. Сведения об аллергиях считаются чувствительной информацией и должны отображаться и передаваться только в контексте бронирования и подготовки класса. |
| Приемка | Юридическое ревью пользовательских текстов согласия и политики; техническая проверка, что аллергии не попадают в логи и не доступны другим клиентам. |
| Обоснование | Требование фиксирует практичный минимум комплаенса без усложнения MVP медицинскими или платежными регуляторными режимами. |

### NFR-13 — Безопасность доступа и сессий

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Доступ к профилю, истории, аллергиям, бронированиям и оценкам разрешен только после успешной авторизации. SMS-код действует не более 5 минут, допускает не более 3 попыток ввода и повторно запрашивается не чаще 1 раза в минуту. При выходе из профиля локальная сессия должна быть завершена, а токен — отозван или признан недействительным на стороне бэкенда. |
| Приемка | Интеграционные тесты авторизации, выхода, истечения SMS-кода, превышения попыток и доступа к профилю без сессии. |
| Обоснование | Эти правила защищают клиентские данные без сложной многофакторной модели, избыточной для MVP. |

### NFR-14 — Поддерживаемые браузеры, устройства и адаптивность

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Поддерживаются последние 2 стабильные версии Google Chrome Desktop, Google Chrome Android и Mozilla Firefox Desktop на дату релиза. Контрольные разрешения: 360×800 и 390×844 для мобильных устройств, 768×1024 для планшетной ширины, 1366×768 и 1920×1080 для desktop. Минимальная ширина корректной работы — 360 px. JavaScript и cookies должны быть включены. |
| Приемка | Кросс-браузерная проверка основных сценариев: вход, просмотр расписания, фильтрация, бронирование, отмена, профиль, выход, оценка шефа. |
| Обоснование | Набор покрывает типичные телефоны и desktop-устройства клиентов без поддержки устаревших браузеров. |

### NFR-15 — Локализация и форматы данных

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Интерфейс MVP должен быть на русском языке. Даты отображаются в формате ДД.ММ.ГГГГ, время — в формате ЧЧ:ММ, часовой пояс — локальный часовой пояс бизнеса, телефон — в российском формате +7 XXX XXX-XX-XX, денежные суммы — в рублях с символом ₽. |
| Приемка | Проверка UI-текстов, сообщений об ошибках, дат, времени, телефона и стоимости на тестовых данных. |
| Обоснование | Единые форматы убирают неоднозначность для клиентов и сотрудников одной точки. |

### NFR-16 — Поведение при ошибках

| Атрибут | Значение |
| --- | --- |
| Приоритет | Must Have |
| Метрика / правило | Все ошибки должны показываться пользователю на русском языке без технических stack trace. При сетевом тайм-ауте чтения система предлагает повторить запрос. При ошибке бронирования система не должна показывать бронь как подтвержденную, пока бэкенд не вернул успешный ответ. При конфликте мест система должна показать, что мест уже нет или данные устарели, и обновить расписание. При ошибке отмены бронь остается в текущем статусе до подтверждения бэкендом. Повторная отправка небезопасных операций должна быть защищена от дублей через блокировку кнопки, idempotency key или эквивалентный механизм. |
| Приемка | Тесты или ручные проверки для тайм-аута, 4xx, 5xx, конфликта мест, повторного клика по кнопке бронирования и недоступности бэкенда. |
| Обоснование | Правила предотвращают двойные брони, ложные подтверждения и непонятные технические сообщения для клиента. |

## Требования к тестовому окружению

| Атрибут | Значение |
| --- | --- |
| Приоритет | Should Have |
| Метрика / правило | Тестовое окружение должно быть отделено от production, использовать отдельную базу данных и отдельные тестовые учетные записи. Оно должно поддерживать мок или sandbox для SMS, тестовый API бэкенда, управляемые тестовые данные расписания, сценарии полной занятости класса, отмены брони, истечения SMS-кода, ошибок 4xx/5xx и сетевого тайм-аута. Тестовые персональные данные должны быть синтетическими. |
| Приемка | Перед релизом на тестовом окружении выполняется smoke-check основных сценариев и проверка негативных сценариев ошибок. |
| Обоснование | Такое окружение достаточно для уверенного релиза MVP и не требует полной копии production с реальными персональными данными. |
