На этом этапе нужен именно структурированный перечень пользовательских историй, отражающих потребности пользователей и ценность, которую система должна им дать.

1. Изучи исходное описание домена.
2. Используй файл `analysis/3-requirements/business-requirements/business-requirements.md` как дополнительный источник контекста.

4. Используй файл `analysis/3-requirements/non-functional-requirements/non-functional-requirements.md` как источник нефункциональных требований.
5. Выдели все пользовательские истории, которые следуют из описания системы, ролей, процессов, сценариев и ограничений.
6. Убери дубли и пересечения.
7. Объедини близкие истории, если они описывают одну пользовательскую потребность и не теряют смысл при объединении.
8. Переформулируй истории в ясном, однозначном и проверяемом виде.
9. Если история сформулирована неявно, но ее можно уверенно вывести из текста, пометь ее как `выведено из контекста`.

11. Не добавляй ничего от себя и не расширяй рамки домена.

## Что считать user story

User story описывает потребность конкретной роли пользователя и ожидаемую ценность.

Формулируй истории в стиле:

* `Как [роль], я хочу [действие/возможность], чтобы [ценность/результат].`

Допускаются и более нейтральные формулировки, если они точнее передают смысл исходного материала, но они должны сохранять пользовательскую перспективу и ценность.

## Что считать good user story

Хорошая user story должна быть:

* ориентированной на пользователя;
* атомарной;
* понятной без дополнительного контекста;
* проверяемой;
* связанной с конкретной потребностью;
* сформулированной без технической воды;
* достаточно маленькой, чтобы быть реализуемой и оцениваемой.

Если исходный материал описывает несколько разных ролей, выделяй истории отдельно для каждой роли, если их потребности различаются.

## Что не включать

Не включай в user stories:

* чисто функциональные формулировки без пользователя и ценности;
* нефункциональные требования;
* бизнес-цели без привязки к пользователю;
* use cases с пошаговыми сценариями;
* детали реализации, архитектуру, технологии;
* повторяющиеся формулировки, не добавляющие нового смысла;
* слишком общие истории вроде `Как пользователь, я хочу пользоваться системой`.

## Как формулировать истории

Каждая история должна отвечать на три вопроса:

* кто пользователь;
* что он хочет сделать;
* зачем ему это нужно.

Формулируй истории так, чтобы они были пригодны для бэклога. Избегай абстрактных фраз. Используй реальные роли из исходного материала: клиент, администратор, менеджер, оператор, модератор, владелец и т. п., если они действительно есть в описании.

Если пользовательская ценность явно не указана, но ее можно уверенно вывести из контекста, укажи это.

## Приоритизация MoSCoW

Правила приоритизации:

* не ставь всем историям `Must Have` по умолчанию;
* опирайся только на исходное описание;
* если данных недостаточно, выбери наиболее вероятный приоритет и добавь примечание;
* если история поддерживает основной сценарий системы, приоритет может быть выше;
* если история относится к вспомогательному или редкому сценарию, не завышай приоритет без оснований.

## Связь с источником

Для каждой user story укажи, откуда она взята:

* прямое указание в исходном описании;
* выведено из функциональных требований;
* выведено из бизнес-правил;
* выведено из процесса или сценария;
* выведено из контекста;
* неясно / требуется уточнение.

Если возможно, добавь краткую ссылку на исходный фрагмент или его смысловую часть.

## Требования к качеству

* Не создавай слишком общие истории вроде `Как пользователь, я хочу пользоваться системой`.
* Не дроби одну историю на слишком мелкие части без необходимости.

* Если в исходном материале присутствуют только сущности без поведения, извлеки только те user stories, которые можно обоснованно вывести из процессов, ограничений, бизнес-правил и функциональных требований.
