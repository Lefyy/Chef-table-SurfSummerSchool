## User Stories

### US-01 — Вход по телефону и SMS-коду

| Атрибут                      | Значение |
| ---------------------------- | -------- |
| Приоритет (MoSCoW)           | Must Have |
| Формулировка                 | Как клиент, я хочу войти в приложение по номеру телефона и одноразовому SMS-коду, чтобы получить доступ к своим бронированиям и сценариям записи. |
| Описание                     | Клиент вводит телефон, получает одноразовый SMS-код и после корректного ввода получает доступ к приложению. |
| Роль                         | Клиент |
| Связь с исходным описанием   | Прямое / Выведено из функциональных требований |
| Источник                     | Описание домена: процесс «Вход / Авторизация»; функциональные требования FR-01, FR-02; бизнес-правило BR-10. |
| Примечания / решения | Код действует 5 минут, лимит ввода — 3 попытки. Интерфейсы администратора, владельца и шефа не входят в текущую поставку. |

## Зафиксированные ограничения и решения

* Возвраты, компенсации и альтернативы при отмене класса студией не предусмотрены в MVP; соответствующие user stories не выделялись.
* Переносы брони, лист ожидания, штрафы, онлайн-оплата, промокоды, подписки, уведомления о новых слотах, лояльность и реферальные программы исключены из MVP.
* Текущая поставка описывает только клиентскую часть; пользовательские сценарии администратора, владельца и шефа не входят в этот объем.
* Дополнительные данные о шефе в деталях класса не нужны; достаточно имени, специализации и среднего рейтинга.
* Итоговая сводка бронирования содержит только выбранный слот, выбранную экипировку и выбранные аллергии.
* Доступ к комментариям к оценке шефа за пределами клиентского приложения не имеет значения для текущей поставки.
* Количественные требования к производительности, нагрузке, срокам хранения данных, резервному копированию и восстановлению указаны в NFR; они не преобразованы в user stories, так как являются нефункциональными или эксплуатационными требованиями.

## Противоречия и дубли

* В исходных материалах статус отмены студией встречается как «Отменён студией» и «Отменена студией»; в user stories использована формулировка «Отменена студией», но термин требует унификации.
* Требования о недопущении двойных броней, наличии свободных мест и атомарном создании брони пересекаются; в user stories они сведены к подтверждению активной брони и отдельному сценарию информирования клиента об отсутствии мест.
* Пуш-уведомления упоминаются в контексте отмены класса студией, но явно исключены из MVP; user story на уведомления не включена.
* Оплата на месте описана как бизнес-правило и граница MVP; выделена только история просмотра статуса оплаты, без онлайн-оплаты и платёжных действий.
* Повторная оценка и изменение оценки шефа запрещены по функциональным требованиям; отдельная user story не выделялась, чтобы не превращать ограничение в самостоятельную пользовательскую потребность.
* Общая обработка ошибок бэкенда сообщением «Ошибка сервера» не выделялась как user story, так как это функциональное/UX-ограничение, а не самостоятельная пользовательская цель.
