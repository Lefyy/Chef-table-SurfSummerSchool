### UC-01 — Авторизация клиента по телефону и SMS-коду

| Атрибут | Значение |
| --- | --- |
| Приоритет (MoSCoW) | Must Have |
| Актор | Клиент |
| Цель | Получить доступ к персональным сценариям приложения: бронированию, профилю, истории, отмене брони и оценке шефа. |
| Триггер | Клиент открывает приложение или переходит к действию, требующему авторизации. |
| Предусловия | Клиент использует поддерживаемый веб-браузер; SMS-шлюз и бэкенд доступны; у клиента есть номер телефона. |
| Основной поток | 1. Клиент вводит номер телефона.<br>2. Система передаёт номер в бэкенд для отправки одноразового SMS-кода.<br>3. Клиент вводит полученный 4-значный код.<br>4. Система передаёт код в бэкенд для проверки.<br>5. При корректном коде, сроке действия не более 5 минут и не превышенном лимите попыток система получает JWT/сессию.<br>6. Система предоставляет клиенту доступ к клиентским функциям приложения. |
| Альтернативные потоки | A1. Если клиент запрашивает повторную отправку кода не ранее чем через 1 минуту и лимит повторных отправок не превышает 5, система инициирует отправку нового SMS-кода.<br>A2. Если повторная отправка запрошена раньше чем через 1 минуту, система не инициирует отправку и оставляет клиента в сценарии авторизации.<br>A3. Если клиент уже авторизован и сессия действительна, система не требует повторного ввода SMS-кода. |
| Исключения | E1. Если код неверный, система не создаёт сессию и учитывает попытку ввода.<br>E2. Если код просрочен, система не создаёт сессию.<br>E3. Если превышен лимит 3 попыток ввода, система не предоставляет доступ в рамках текущей проверки кода.<br>E4. Если превышен лимит 5 повторных отправок SMS-кода, система не отправляет новый код в рамках текущего сценария авторизации.<br>E5. При ошибке бэкенда без специального описания система показывает сообщение «Ошибка сервера». |
| Постусловия | Клиент авторизован и имеет активную сессию; при неуспехе сессия не создаётся. |
| Формулировка сценария | Клиент авторизуется в приложении по номеру телефона и одноразовому SMS-коду, чтобы получить доступ к клиентским сценариям. |
| Связь с исходным описанием | Прямое / Выведено из функциональных требований / Выведено из user stories / Выведено из нефункциональных требований |
| Источник | Описание домена: процесс «Вход / Авторизация», BR-10; FR-01, FR-02, FR-03; US-01, US-02; NFR-04, NFR-05, NFR-06. |
| Примечания / неоднозначности | Интерфейсы администратора, владельца и шефа не входят в текущую поставку. Количественные требования к времени отклика авторизации не указаны в исходном описании. |

### UC-02 — Просмотр и фильтрация расписания классов

| Атрибут | Значение |
| --- | --- |
| Приоритет (MoSCoW) | Must Have |
| Актор | Клиент |
| Цель | Найти подходящий кулинарный класс для записи. |
| Триггер | Клиент открывает экран расписания или изменяет параметры фильтрации. |
| Предусловия | Бэкенд предоставляет данные о слотах, программах, шефах, свободных местах и прокате; клиент использует поддерживаемый браузер. Авторизация явно не указана как обязательная для просмотра расписания. |
| Основной поток | 1. Клиент открывает расписание.<br>2. Система получает из бэкенда доступные слоты и по умолчанию показывает классы на ближайшие 7 календарных дней.<br>3. Система отображает для каждого слота время, программу, шефа, количество свободных мест, признак духовки и признак доступности проката.<br>4. Клиент задаёт фильтры по дате или диапазону дат, программе/меню, шефу, уровню сложности, наличию свободных мест и/или доступности проката.<br>5. Система применяет выбранные фильтры совместно по И-логике.<br>6. Система показывает только классы, соответствующие всем выбранным условиям. |
| Альтернативные потоки | A1. Если расписание или результат фильтрации пустой, система показывает пустое состояние «Пока нет доступных классов».<br>A2. Если клиент сбрасывает фильтры, система возвращает отображение расписания без пользовательских условий фильтрации в пределах доступных данных. |
| Исключения | E1. При ошибке бэкенда без специального описания система показывает сообщение «Ошибка сервера». |
| Постусловия | Клиент видит список подходящих классов или сообщение об отсутствии доступных классов. |
| Формулировка сценария | Клиент просматривает расписание и применяет фильтры, чтобы выбрать подходящий слот для дальнейшего просмотра и бронирования. |
| Связь с исходным описанием | Прямое / Выведено из функциональных требований / Выведено из user stories / Выведено из бизнес-правил / Выведено из процесса |
| Источник | Описание домена: процесс «Просмотр и фильтрация расписания», карточка слота, пустое состояние, BR-09; BR-07, BR-08; FR-04–FR-09; US-03–US-06; NFR-14. |
| Примечания / неоднозначности | Более длинный период через фильтр дат упомянут в бизнес-требованиях, но пределы такого периода в исходном описании не указаны. Количественные показатели производительности фильтрации не указаны. |

Клиент принимает решение перейти к бронированию или вернуться к расписанию. |

### UC-04 — Создание брони на выбранный класс

| Атрибут | Значение |
| --- | --- |
| Приоритет (MoSCoW) | Must Have |
| Актор | Клиент |
| Цель | Самостоятельно записаться на доступный кулинарный класс. |
| Триггер | Клиент выбирает действие перехода к бронированию из деталей слота. |
| Предусловия | Клиент авторизован; выбранный слот существует; слот не отменён студией; `availableSeats > 0`; бэкенд доступен и является источником истины; онлайн-оплата не требуется, так как оплата производится на месте. |
| Основной поток | 1. Клиент запускает бронирование выбранного слота.<br>2. Система показывает доступные позиции проката экипировки, полученные от бэкенда.<br>3. Клиент выбирает ноль, одну или несколько доступных позиций проката.<br>4. Система показывает справочник аллергенов и, если есть сохранённые аллергии клиента, подставляет их для новой брони.<br>5. Клиент указывает ноль, одну или несколько аллергий из справочника.<br>6. Система показывает итоговую сводку выбранного слота, проката и аллергий.<br>7. Клиент подтверждает бронь.<br>8. Система передаёт данные подтверждённой брони в бэкенд.<br>9. Бэкенд атомарно создаёт бронь, а система отображает бронь со статусом «Активна».<br>10. Система сохраняет выбранные аллергии в профиле клиента для последующих бронирований. |
| Альтернативные потоки | A1. Клиент не выбирает прокат; система продолжает бронирование без позиций проката.<br>A2. Клиент не указывает аллергии; система продолжает бронирование без аллергий.<br>A3. Если позиция проката имеет `stockAvailable = 0`, система показывает её как недоступную для выбора или не допускает выбор этой позиции.<br>A4. Если клиент изменяет выбранные аллергии в процессе бронирования, новый набор используется в создаваемой брони и сохраняется в профиле.<br>A5. Если клиент отказывается от подтверждения на сводке, система не отправляет запрос на создание брони. |
| Исключения | E1. Если при подтверждении `availableSeats = 0` или бэкенд отказал из-за отсутствия мест, система не создаёт бронь, показывает ошибку отсутствия мест и переносит клиента на страницу листинга классов.<br>E2. Если слот отменён студией, система запрещает создание брони на этот слот.<br>E3. Если клиент пытается указать аллергию вне справочника, система не включает её в бронь.<br>E4. При ошибке бэкенда без специального описания система показывает сообщение «Ошибка сервера». |
| Постусловия | При успехе создана активная бронь, место закреплено за клиентом, выбранный прокат передан в бронь, аллергии сохранены в профиле; при неуспехе бронь не создаётся. |
| Формулировка сценария | Клиент проходит воронку бронирования: выбирает прокат и аллергии, проверяет сводку и подтверждает создание активной брони через бэкенд. |
| Связь с исходным описанием | Прямое / Выведено из функциональных требований / Выведено из user stories / Выведено из бизнес-правил / Выведено из процесса |
| Источник | Описание домена: процесс «Воронка бронирования», сущности Booking, RentalItem, Allergen, BR-01, BR-03, BR-05, BR-06, BR-11, BR-12; BR-01, BR-03–BR-05, BR-09, BR-10, BR-13, BR-14; FR-11–FR-18, FR-22, FR-30; US-08–US-14, US-18; NFR-09, NFR-19. |
| Примечания / неоднозначности | Атомарность бронирования и списание проката обеспечиваются бэкендом. Количество мест в одной брони явно не указано; из описания уверенно следует только запись клиента на класс. |

### UC-05 — Отмена активной брони клиентом

| Атрибут | Значение |
| --- | --- |
| Приоритет (MoSCoW) | Must Have |
| Актор | Клиент |
| Цель | Отменить свою активную бронь заранее, если клиент не сможет прийти. |
| Триггер | Клиент выбирает отмену активной брони в своих бронированиях. |
| Предусловия | Клиент авторизован; у клиента есть активная бронь; до начала класса осталось более 12 часов; бэкенд доступен. |
| Основной поток | 1. Клиент открывает свою активную бронь.<br>2. Клиент выбирает действие отмены.<br>3. Система проверяет ограничение по времени начала класса на основании данных бэкенда.<br>4. Если до начала класса более 12 часов, система передаёт запрос отмены в бэкенд.<br>5. Бэкенд отменяет бронь и освобождает место.<br>6. Система отображает бронь со статусом «Отменена клиентом». |
| Альтернативные потоки | A1. Если до начала класса осталось 12 часов или меньше, система не выполняет отмену и показывает сообщение «Отмена недоступна менее чем за 12 часов до класса». |
| Исключения | E1. Если бронь уже не активна, система не выполняет отмену в рамках этого сценария.<br>E2. При ошибке бэкенда без специального описания система показывает сообщение «Ошибка сервера». |
| Постусловия | При разрешённой отмене бронь получает статус «Отменена клиентом», место возвращается; при поздней отмене бронь остаётся без изменений. |
| Формулировка сценария | Клиент отменяет активную бронь более чем за 12 часов до начала класса через приложение. |
| Связь с исходным описанием | Прямое / Выведено из функциональных требований / Выведено из user stories / Выведено из бизнес-правил |
| Источник | Описание домена: процесс «Отмена брони клиентом», BR-04; BR-11; FR-19, FR-20; US-15, US-16. |
| Примечания / неоднозначности | Штрафы, переносы, waitlist, возвраты и компенсации исключены из MVP. Зависит от корректного времени начала класса, полученного из бэкенда. |

### UC-06 — Просмотр отмены класса студией в бронировании

| Атрибут | Значение |
| --- | --- |
| Приоритет (MoSCoW) | Must Have |
| Актор | Клиент |
| Цель | Узнать, что класс отменён студией, и увидеть причину отмены. |
| Триггер | Клиент открывает историю или данные брони, класс по которой был отменён студией во внешней админке. |
| Предусловия | Клиент авторизован; у клиента есть бронь на класс; отмена класса выполнена во внешней административной системе; бэкенд вернул статус отмены студией и причину. |
| Основной поток | 1. Клиент открывает профиль или историю бронирований.<br>2. Система получает данные броней из бэкенда.<br>3. Для брони по отменённому студией классу система отображает статус «Отменена студией».<br>4. Система отображает причину отмены, полученную от бэкенда.<br>5. Система не предлагает повторную запись на отменённый студией слот. |
| Альтернативные потоки | A1. Если клиент пытается открыть отменённый студией слот из расписания или истории, система не допускает создание новой брони на этот слот. |
| Исключения | E1. Если причина отмены не получена от бэкенда, порядок отображения причины не указано в исходном описании.<br>E2. При ошибке бэкенда без специального описания система показывает сообщение «Ошибка сервера». |
| Постусловия | Клиент видит актуальный статус и причину отмены; повторная запись на отменённый студией слот недоступна. |
| Формулировка сценария | Клиент просматривает бронь по классу, отменённому студией, и получает информацию о статусе и причине отмены без возможности повторной записи на этот слот. |
| Связь с исходным описанием | Прямое / Выведено из функциональных требований / Выведено из user stories / Выведено из бизнес-правил / Выведено из процесса |
| Источник | Описание домена: процесс «Отмена класса студией», BR-11; BR-12, BR-13; FR-21, FR-22; US-17, US-18; NFR-18. |
| Примечания / неоднозначности | Отмена выполняется не в клиентском приложении. Push-уведомления не входят в MVP. В источниках есть разные формулировки статуса: «Отменён студией» и «Отменена студией». |

### UC-07 — Просмотр истории бронирований и статуса оплаты

| Атрибут | Значение |
| --- | --- |
| Приоритет (MoSCoW) | Must Have |
| Актор | Клиент |
| Цель | Просмотреть свои прошедшие, текущие и предстоящие бронирования и понимать статус каждой записи. |
| Триггер | Клиент открывает вкладку «История» в профиле. |
| Предусловия | Клиент авторизован; бэкенд предоставляет историю бронирований клиента. |
| Основной поток | 1. Клиент открывает профиль.<br>2. Клиент переходит на вкладку «История».<br>3. Система запрашивает историю бронирований клиента у бэкенда.<br>4. Система отображает брони клиента со статусом класса для клиента: «Прошёл», «Идёт» или «Предстоит».<br>5. Система отображает по каждой брони выбранные аллергии, экипировку, шефа, дату и время класса.<br>6. Система отображает статус оплаты брони со значением «Не оплачено» или «Оплачено», если такой статус получен от бэкенда. |
| Альтернативные потоки | A1. Если исторических данных нет, система отображает пустое состояние истории; исторические данные до запуска системы отсутствуют по допущению. |
| Исключения | E1. Если клиент не авторизован, доступ к персональной истории не предоставляется.<br>E2. При ошибке бэкенда без специального описания система показывает сообщение «Ошибка сервера». |

## Неясности и пробелы в исходном описании

* Не указано, обязательна ли авторизация для просмотра расписания и деталей класса; она явно необходима для персональных сценариев бронирования, профиля, отмены и оценки.
* Не указано, сколько мест может бронировать клиент в одной брони; в use cases отражена только запись клиента на класс.
* Не указаны пределы периода фильтрации дат за рамками расписания на ближайшие 7 дней по умолчанию.
* Не указано поведение при отсутствии причины отмены класса студией в данных бэкенда.
* Не указано поведение локальной сессии при ошибке отзыва токена во время выхода из системы.
* Не указано, кто кроме клиента и бэкенда имеет доступ к непубличным комментариям к оценке шефа.
* Не указаны количественные показатели производительности, доступности и масштабируемости для сценариев авторизации, расписания, бронирования, отмены и оценки.
* Не указаны правила оплаты на месте вне отображения статуса оплаты «Не оплачено» или «Оплачено».
* Не указаны правила возвратов, компенсаций, штрафов, переносов и листа ожидания; они исключены из MVP или не описаны.
* Не указано, как студия использует сведения об аллергиях после сохранения, кроме их учёта до класса.

## Противоречия и дубли

* В исходных материалах встречаются разные формулировки статуса отмены студией: «Отменён студией» и «Отменена студией»; в use cases используется «Отменена студией» как формулировка из функциональных требований.
* Push-уведомление при отмене класса студией упоминается в исходном контексте, но затем явно исключено из MVP; в use cases уведомления не включены.
* Требования об отсутствии двойных броней, бронировании только при наличии мест и атомарном создании брони пересекаются; в use cases они объединены в сценарий создания брони и его исключения.
* User stories про выбор проката, указание аллергий, сводку и подтверждение брони описывают этапы одного пользовательского результата; они объединены в UC-04, чтобы не дробить воронку бронирования без необходимости.
* User stories про отмену заранее и запрет поздней отмены объединены в UC-05 как основной и альтернативный поток одного сценария отмены.
* Отображение отмены студией и запрет повторной записи на отменённый слот объединены в UC-06, так как оба поведения связаны с одним состоянием слота/брони.
