/// Нейросеть - Codex (Cloud версия бесплатно).

Ты — опытный AI Business Analyst с экспертизой в UX/UI и Product Discovery.

Проанализируй следующие документы:

* `analysis/2-elicitation/domain-description.md`
* `analysis/3-requirements/business-requirements/business-requirements.md`
* `analysis/3-requirements/functional-requirements/functional-requirements.md`
* `analysis/3-requirements/non-functional-requirements/non-functional-requirements.md`
* `analysis/3-requirements/user-stories/user-stories.md`
* `analysis/3-requirements/use-cases/use-cases.md`

Твоя задача — выявить **все требования, влияющие на UI/UX**, и подготовить документацию для UX/UI-дизайнера.

Не придумывай функциональность. Если решение логически следует из требований, но явно не описано, помечай его как **Inferred**. Если информации недостаточно — помечай как **Missing**.

## Результат

Создай следующую структуру:

```text
design/

├── design-brief.md
│
├── screens/
│   ├── screen-01-<name>.md
│   ├── screen-02-<name>.md
│   └── ...
│
└── bottom-sheets/
    ├── bottom-sheet-01-<name>.md
    ├── bottom-sheet-02-<name>.md
    └── ...
```

Если Bottom Sheet отсутствуют, не создавай папку `bottom-sheets`.

---

## design-brief.md

Документ должен содержать:

* краткое описание продукта;
* цели продукта;
* целевые пользователи и роли;
* основные пользовательские сценарии;
* список всех экранов;
* список всех Bottom Sheets;
* основные UX-требования;
* ограничения и нефункциональные требования, влияющие на интерфейс;
* открытые вопросы;
* assumptions (`Inferred`);
* отсутствующие требования (`Missing`).

---

## Документ экрана

Для **каждого** экрана создай отдельный Markdown-файл.

Используй следующий шаблон:

```md
# <Название экрана>

## Назначение

## Пользовательские роли

## Источники требований

## Когда отображается

## Откуда можно перейти

## Куда можно перейти

## Основной пользовательский сценарий

## Состав экрана

## Действия пользователя

## Связанные Bottom Sheets

## Состояния

- Default
- Loading
- Empty
- Error
- Success
- Disabled
- Offline (если применимо)

## Валидация

## Ограничения

## Edge Cases

## Inferred

## Missing
```

## Документ Bottom Sheet

Для **каждого** Bottom Sheet создай отдельный Markdown-файл.

Используй шаблон:

```md
# <Название Bottom Sheet>

## Назначение

## Когда открывается

## Откуда вызывается

## Содержимое

## Действия пользователя

## Состояния

## Валидация

## Edge Cases

## Связанные экраны

## Inferred

## Missing
```

---

## Требования к анализу

* Выяви **все** экраны, включая промежуточные, служебные и системные.
* Выяви все Bottom Sheets, даже если они прямо не названы, но логически необходимы для пользовательского сценария (помечай как `Inferred`).
* Для каждого элемента указывай, из каких требований или пользовательских сценариев он следует.
* Не объединяй разные экраны в один документ.
* Один экран — один Markdown-файл.
* Один Bottom Sheet — один Markdown-файл.
* Итоговая документация должна быть достаточной для начала проектирования интерфейсов без повторного анализа исходных требований.
